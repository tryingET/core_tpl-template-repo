### softwareco-agents is NOT where agents live

It's a **policy/documentation repo** about how agents operate:
- Agent roles and scopes
- Operating contracts
- Registry of existing agents

Generated agents go to `agents/agent-<slug>` per lane policy.

### Lane Definitions

| Lane | Use For | Template | Key Characteristics |
|------|---------|----------|---------------------|
| **agent** | AI agents | `tpl-agent-repo` | Persona + learnings + safety guardrails |
| **owned** | Own products | `tpl-owned-repo` | High governance, release automation, full stack |
| **contrib** | Upstream contributions | `tpl-contrib-repo` | Light governance, upstream sync focus |
| **infra** | Infrastructure/DevOps | `tpl-infra-repo` | Runbooks + incidents + rollout + governance |
| **org** | Company handbook | `tpl-org-repo` | Governance docs, policies |

### Lane Mapping: holdingco ↔ softwareco

| holdingco | softwareco | Notes |
|-----------|------------|-------|
| `agent` | `agent` | Same ✅ |
| `org` | `org` | Same ✅ |
| `project` | `owned` | Similar - delivery projects |
| - | `contrib` | softwareco-specific (upstream) |
| - | `infra` | softwareco-specific (DevOps) |

**Decision:** L0 includes `tpl-agent-repo`, `tpl-org-repo`, `tpl-project-repo`. Softwareco's `contrib` and `infra` are domain-specific, keep them separate.

## Architecture: Two Patterns Coexist

| Pattern | L1 Style | Templates Live In | Best For |
|---------|----------|-------------------|----------|
| **Monolithic** | Embedded templates | `L1/copier/tpl-*-repo/` | Small companies, single team (holdingco, healthco) |
| **Registry** | Policy + registry | Separate L2 repos | Large orgs, multiple teams (softwareco) |

### L0 Supports Both (Option 3)

1. **Generates monolithic L1** with embedded `tpl-*-repo/` templates
2. **Provides tpl-* templates** that can be copied for registry-style setups

This gives single source of truth with maximum flexibility.
