## Critical Architecture Bug Discovered

### The Problem

**L0 does NOT generate archetype templates.** Currently:

```
L0: core/tpl-template-repo
    └── copier-template/           ← Only generates L1 shell
        └── copier.yml             ← repo_archetype param (trims at runtime)

L1: holdingco-templates (HAND-CRAFTED, not from L0 ❌)
    └── copier/tpl-agent-repo/     ← Manually created
    └── copier/tpl-org-repo/       ← Manually created
    └── copier/tpl-project-repo/   ← Manually created
```

L0 generates `copier/template-repo/` (generic) while holdingco has `copier/tpl-{agent,org,project}-repo/` (purpose-built).

### What It Should Be

```
L0: core/tpl-template-repo
    └── copier-template/
        └── copier/
            ├── tpl-agent-repo/     ← Agent template (embedded in L0)
            ├── tpl-org-repo/       ← Org template (embedded in L0)
            └── tpl-project-repo/   ← Project template (embedded in L0)

L1: healthco-templates (GENERATED from L0 ✅)
    └── copier/
        ├── tpl-agent-repo/         ← Generated by L0
        ├── tpl-org-repo/           ← Generated by L0
        └── tpl-project-repo/       ← Generated by L0
```

### Fix Required (Option A)

1. Copy `holdingco-templates/copier/tpl-agent-repo/` → `L0/copier-template/copier/tpl-agent-repo/`
2. Same for `tpl-org-repo`, `tpl-project-repo`
3. Update L0's copier.yml to include these in generation
4. Regenerate healthco-templates from updated L0

```
├── infra/                    ← Generated "infra" projects
│
├── tpl-agent-repo/           ← L2 template (separate repo)
├── tpl-owned-repo/           ← L2 template for owned lane
├── tpl-contrib-repo/         ← L2 template for contrib lane
├── tpl-infra-repo/           ← L2 template for infra lane
└── tpl-org-repo/             ← L2 template for org lane
```

### Lane Definitions (from lane-policy-matrix.md)

| Lane | Use For | Template | Key Characteristics |
|------|---------|----------|---------------------|
| **owned** | SoftwareCo's own products | `tpl-owned-repo` | High governance, release automation, full stack |
| **contrib** | Upstream contributions | `tpl-contrib-repo` | Light governance, upstream sync focus |
| **agent** | AI agents | `tpl-agent-repo` | Persona + learnings + safety guardrails |
| **infra** | Infrastructure/DevOps | `tpl-infra-repo` | Runbooks + incidents + rollout + governance |

### Template Contents

| Template | Has `src/` | Has `tests/` | Has `governance/` | Special |
|----------|-----------|--------------|-------------------|---------|
| `tpl-owned-repo` | ✅ | ✅ | ❌ | `convex/`, `package.json` (full stack) |
| `tpl-contrib-repo` | ✅ | ✅ | ❌ | Light, upstream-focused |
| `tpl-infra-repo` | ❌ | ❌ | ✅ | Runbooks, incidents |
| `tpl-agent-repo` | ❌ | ❌ | ❌ | `docs/person/`, `prompts/` |

### Key Insight: softwareco-agents is NOT where agents live

It's a **policy/documentation repo** about how agents operate in softwareco:
- Agent roles and scopes
- Operating contracts
- Registry of existing agents

Generated agents would go to `agents/agent-<slug>` per the lane policy.

```

```

3. Allow copying individual tpl-* templates for registry-style setups

This gives maximum flexibility with single source of truth.
